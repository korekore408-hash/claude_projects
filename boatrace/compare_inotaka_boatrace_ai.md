# 外部サイト比較: inotaka1979/boatrace-ai vs 自分の boatrace

対象: https://inotaka1979.github.io/boatrace-ai/ （リポジトリ inotaka1979/boatrace-ai）
解析日: 2026-06-27 / ブランチ: analysis/inotaka-boatrace-ai

調査方法: 公開リポジトリの workflows / scripts / README を直接参照して確認した事実ベース。

---

## 1. 予想ソースの違い

| 観点 | 自分(boatrace) | inotaka(boatrace-ai) |
|---|---|---|
| データ取得元 | **公式LZHを直接DL**（番組表B・結果K を lhafile で解凍） | **非公式 OpenAPI**（`boatraceopenapi.github.io/programs/v2/today.json`）＋**公式サイトHTMLスクレイピング**（結果/オッズ/展示beforeinfo/潮汐tide/選手写真）のハイブリッド |
| 予想モデル | **純Python独自ML 1本**（sklearn不使用）。p_win確率→本命/穴確率 | **9要素スコアリング（ルールベース加重）が本番**。加えて **GBDT（勾配ブースティング）学習基盤**を用意済み（`train_gbdt.py`）だが **schedule は無効化**＝データ蓄積待ちで現状は手動のみ |
| 特徴量の幅 | 選手履歴・レース相対中心（展示は検証したが本番非反映） | 展示・潮汐・選手写真まで取得＝**特徴量の裾野が広い** |

要点: 自分は「公式生データ＋単一独自モデル」で精度を詰める方向。inotaka は「取りやすい非公式API＋広く浅く特徴量を集め、ルールスコア＋将来GBDT」の拡張志向。

---

## 2. 回収率（ROI）ソースの違い

| 観点 | 自分 | inotaka |
|---|---|---|
| 位置づけ | **オフライン研究**。バックテストで「EV/オッズにエッジなし(72〜77%)」「勝者の呪い」と**結論済み** | **フロントのインタラクティブ・バックテスタ**。EV閾値1.00〜1.50 / Kelly比0〜1.0 / 期間7・14・30日・全履歴 を画面で変えて ROI を即時評価（Forward-chain評価即時実行） |
| 対象買い目 | 本命/穴の分岐・買目点数最適化 | 3連単・2連単・**穴狙い**を戦略として評価 |
| 拡張 | 単独で検証 | **連合学習(FL)** でコミュニティの重みを集約（GitHub Issue 経由でユーザー貢献を受付） |
| 数値公開 | 検証結論として明示 | **具体的な回収率%は非表示**（"参考情報・的中保証なし"のディスクレーマ） |

要点: 自分は「勝てるエッジは無い」という**負の研究結論**を持っている。inotaka は「ユーザーが戦略を試せるツール」志向で、回収率は固定の主張ではなくパラメータ次第で動かす設計。どちらも具体回収率の断定は避けている＝姿勢は近い。

## 3. 更新方法の違い

| 観点 | 自分 | inotaka |
|---|---|---|
| 配信 | Cloudflare Pages + **パスワードゲート(非公開)** | **GitHub Pages 公開** + Cloudflare Worker(エッジ取得) + **PWA**(ホーム追加) |
| 生成/更新 | daily 1本(04:17 JST) + **2時間毎 update**(update.json) | **scraperを種類別に多数分割**（results/odds/racedata/previews/tide）+ `refresh-next-open`(**1日5回** cron: 09:15/10:20/13:20/17:20/21:15 JST) + `build-db`(毎日) |
| 監視/堅牢化 | なし（シンプル運用） | `data-freshness-monitor`(**5分毎**に鮮度監視→**stale検知で scrape-all を自動起動**) + `auto-rollback` + Lighthouse/e2e/gate |
| GitHub cron制約への対処 | 2時間毎の単一ジョブで足りる | コメントに「GitHubは1 workflowあたり schedule を4-6回/日に**間引く**」と実測記載→**多workflow分割＋鮮度監視で自己起動**して間引きを回避 |

要点: 自分は**単純・低頻度（運用コスト最小）**。inotaka は**多層・高頻度＋自動監視/ロールバック**で、GitHubのスケジュール間引きにも自己修復で対抗する本格運用。

## まとめ（自分への示唆）
- データ源: inotaka の **非公式OpenAPI(today.json)** はLZH解凍不要で軽い。取得の簡便化の選択肢になる（ただし非公式＝可用性リスク）。
- モデル: 自分の「エッジ無し」結論は強い。inotaka も回収率を断定せず＝方向性は一致。GBDT化は inotaka でも"データ蓄積待ち"で未稼働。
- 更新: 自分の2時間毎単一ジョブは十分シンプル。ただし inotaka の **鮮度監視→自動再取得** の発想は、cron間引きで更新が抜けた今回のような事故の保険になり得る。
